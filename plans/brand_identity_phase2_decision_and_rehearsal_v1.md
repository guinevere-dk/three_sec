## 1) 2-1 변경 필요성 판단 (결론)

### 최종 결론

- Android `applicationId`: **유지** (`com.dk.three_sec`)
- iOS `PRODUCT_BUNDLE_IDENTIFIER`: **유지** (`com.dk.three_sec`)
- 결제 상품 ID(`3s_*`): **유지**

### 근거 요약

1. 패키지/번들 식별자 변경 시 스토어 배포 라인이 분기되어 신규 앱 취급 리스크가 크다.
2. Firebase 앱 등록(안드/ios), OAuth 콘솔, 푸시/딥링크 연동 재설정이 동반되어 장애면이 넓다.
3. 결제 상품 ID를 변경하면 기존 구독 연속성 리스크가 높다.
4. 현재 목표는 브랜드명(사용자 노출) 정렬이며, 기술 식별자까지 동시 변경할 사업적 이득보다 운영 리스크가 더 크다.

### 차기 검토 조건(변경 트리거)

- 신규 앱 전략(국가/세그먼트 분리 출시)으로 의도적 앱 분리 필요
- 법무/상표 이슈로 기술 식별자 강제 변경 필요
- 스토어/결제/인증 전환 비용 대비 명확한 사업 KPI 개선 근거 확보

## 2) 2-3 마이그레이션 리허설 계획 (식별자 변경 시에만 실행)

## 2-3-1. 스테이징 구성

- Android 스테이징 식별자 예시: `com.dk.onesecondvlog.stg`
- iOS 스테이징 식별자 예시: `com.dk.onesecondvlog.stg`
- Firebase: 스테이징 프로젝트 별도 분리(프로덕션 데이터와 격리)
- Social Exchange 함수: 스테이징 엔드포인트 분리

## 2-3-2. E2E 검증 시나리오

1. 로그인
   - Google / Apple / Kakao / Naver 로그인
   - 실패 시 에러 코드 및 UX 메시지 확인

2. 결제
   - `3s_*` SKU 조회 성공
   - 구매/복원/해지 동작 및 Firestore 반영 확인

3. 푸시
   - 권한 요청 → 토픽 구독 → 푸시 수신

4. 데이터 접근
   - 동일 사용자 UID 기준 Firestore/Storage 접근 성공
   - 세션 전환(로그아웃/재로그인) 이후 동기화 무결성 확인

5. 딥링크/OAuth 콜백
   - Android intent-filter 및 iOS URL scheme 콜백 정상 동작

## 2-3-3. 합격 기준(Go/No-Go)

- 핵심 여정(로그인/결제/푸시/딥링크) 성공률 100%
- P0 결함 0건, P1 결함 0건
- 24시간 모니터링에서 로그인 실패율/결제 실패율/치명 크래시 임계치 이탈 없음

## 3) 롤백 시나리오 (당일 복구 포함)

## 3-1. 롤백 트리거

- 로그인 실패율 급증(임계치 초과)
- 결제 검증 실패 또는 SKU 미노출
- OAuth 콜백 실패 다발
- 치명 크래시율 급증

## 3-2. 당일 복구 절차

1. 배포 중단
   - 프로덕션 확산 중지(내부/클로즈드/프로덕션 트랙 추가 승격 중단)

2. 기능 플래그/엔드포인트 복귀
   - `SOCIAL_AUTH_EXCHANGE_URL` 스테이징/신규 경로를 즉시 기존 안정 경로로 환원

3. 앱 빌드 라인 복귀
   - 기존 식별자 기반 빌드로 핫픽스 배포(버전코드 상향)

4. 콘솔 설정 롤백
   - OAuth 공급자 콘솔 redirect/scheme 값을 기존 앱 식별자로 복귀
   - Firebase 앱 설정/키 매핑을 기존 안정 세트로 복귀

5. 사후 검증
   - 로그인/결제/푸시 smoke 테스트 통과 확인
   - 장애 시간대 사용자 영향도 집계 및 CS 공지 반영

## 3-3. 롤백 준비물 체크리스트

- 기존 안정 빌드 재배포 가능한 브랜치/태그
- 기존 OAuth 콘솔 값 백업본
- Firebase 앱 설정값/키 백업본
- 운영 공지/CS 템플릿
