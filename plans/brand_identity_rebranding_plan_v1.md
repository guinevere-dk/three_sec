# 브랜드명 개편 추진 계획서 v1

작성일: 2026-03-30  
대상: `three_sec_vlog` 앱(Flutter + Android/iOS + Firebase + 결제 연동)

---

## 0) 목적

- 길이 정책(1초 전환)과 **브랜드명 개편 작업을 분리**해 릴리즈 리스크를 낮춘다.
- 사용자 노출 명칭, 기술 식별자(패키지/번들/Firebase), 상점 메타데이터를 단계적으로 정리한다.
- 결제/인증/배포 파이프라인에 영향을 주는 변경을 사전 검증 가능한 형태로 추진한다.

---

## 1) 변경 범위 정의

## A. 사용자 노출 레이어(우선)

- 앱 이름(표시명)
- 앱 내 카피(온보딩/로그인/공지/프로필/약관 요약)
- 스토어 소개 문구(Play/App Store)
- 고객 커뮤니케이션(릴리즈 노트, FAQ)

## B. 기술 식별자 레이어(신중)

- Android `applicationId` (예: `com.dk.three_sec`)
- iOS `PRODUCT_BUNDLE_IDENTIFIER`
- Firebase 프로젝트명/앱 등록 정보
- OAuth Redirect URI/딥링크 스킴
- 결제 상품 ID(`3s_*`) 및 서버 검증 규칙

## C. 자산/운영 레이어

- 앱 아이콘/스플래시/브랜드 에셋
- 개인정보처리방침/이용약관 브랜드 표기
- 모니터링 대시보드/로그 태그/운영 문서 명칭

---

## 2) 핵심 원칙

1. **1차는 사용자 노출명 우선**
   - 식별자/상품 ID는 즉시 변경하지 않고, 사용자 인지부터 정렬한다.

2. **식별자 변경은 별도 릴리즈 트랙으로 분리**
   - 패키지/번들 변경은 스토어 상 신규 앱 취급 가능성이 있어 단독 검토한다.

3. **결제 ID는 유지 우선**
   - 구독 상품 ID 변경은 기존 구독자에 영향이 크므로 가능하면 유지하고 표시명만 개편한다.

4. **백엔드 연동 키는 사전 매트릭스로 검증**
   - 인증, Dynamic Link, 푸시, 결제 검증 키를 변경 전후 비교표로 통제한다.

## 3) 단계별 실행 계획

## Phase 1. 브랜드 메시지/표시명 정렬 (저위험)

### 1-1. 명칭 정책 확정
- 최종 브랜드명(국문/영문)
- 약칭, 표기 가이드(띄어쓰기/대소문자)
- 금지 표기(구브랜드 혼용 규칙)

#### 확정안 (2026-03-30)
- 국문 브랜드명: **원세컨 브이로그**
- 영문 브랜드명: **One Second Vlog**
- 약칭: **1s Vlog**

#### 표기 가이드
- 사용자 노출 영역에서 기본 표기는 국문 **원세컨 브이로그**를 우선 사용한다.
- 영문 표기가 필요한 영역(스토어/웹 메타/이메일 제목 등)은 **One Second Vlog**를 사용한다.
- 짧은 라벨/좁은 UI 영역은 약칭 **1s Vlog**를 사용한다.

#### 금지 표기
- `3s`
- `Three Sec Vlog`
- `3-Second Vlog`
- `three_sec_vlog` (사용자 노출 맥락)

### 1-2. 앱 내 노출 텍스트 일괄 수정
- 대상 파일(예시)
  - `lib/main.dart`
  - `lib/screens/login_screen.dart`
  - `lib/screens/announcements_screen.dart`
  - `README.md`, `plans/*.md` 중 사용자 전달 문구

### 1-3. 스토어 메타데이터 반영
- 앱 이름, 부제, 설명, 스크린샷 카피 동기화
- 릴리즈 노트에 "브랜드명 개편" 공지

#### 릴리즈 노트 문구(권장)
- `브랜드명을 3s에서 원세컨 브이로그(One Second Vlog)로 개편했습니다. 기능/계정/결제는 기존과 동일하게 유지됩니다.`

**DoD**
- 앱 실행~핵심 화면에서 구브랜드 표기가 노출되지 않음
- 스토어/앱 내/문서의 사용자 노출명이 동일

## 4) 리스크 및 대응

1. **스토어 식별자 변경으로 앱 분리 위험**
   - 대응: 식별자 유지 전략 우선, 필요 시 신규 앱 전략 문서 별도 수립

2. **결제 상품 ID 변경에 따른 구독 단절 위험**
   - 대응: 상품 ID 유지 + 노출명만 변경, 서버 매핑 테이블 유지

3. **OAuth/딥링크 불일치로 로그인 실패 위험**
   - 대응: 공급자 콘솔 설정 체크리스트 + 사전 자동화 테스트

4. **문구 혼재로 사용자 혼란**
   - 대응: 금지어 리스트 기반 전수 검색/검수 절차 운영
