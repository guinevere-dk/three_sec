# 외부 이미지/영상 다중 가져오기 개선 계획 v1

## 1) 목표
- 외부 미디어 가져오기에서 **한 번에 여러 개 선택** 가능하도록 개선한다.
- **이미지 선택 시 즉시 1.5초 영상으로 변환**한다.
- 사용자 노출 문구는 기존대로 `1s`를 유지한다.
- **영상 선택 시 순차 처리(큐 기반)**로 안정적으로 진행한다.
- 기존 단일 선택 동작과 튜토리얼/앨범 흐름을 깨지 않고 확장한다.

### 정책 가정(실제/노출 분리)
- 실제 처리 길이 목표: `1500ms`
- 노출 길이 문구: `1s` 유지
- 허용 편차: `1500ms` 중심, 플랫폼별 동적 오차 허용
- 실패/집계/요약 수치 계산은 실제 길이(1500ms)와 표시(`1s`)를 섞지 않음

## 2) 현재 기준점
- 외부 가져오기 진입 로직: `lib/main.dart`의 `_pickMedia()`
- 현재 동작:
  - `ImagePicker.pickMedia()`로 단일 파일 선택
  - 영상이면 `ClipExtractorScreen`으로 이동
  - 이미지면 `convertPhotoToVideo()` 즉시 실행

## 3) 요구사항 정리

### 기능 요구사항
1. 사용자 외부 가져오기 액션 시 다중 선택을 허용한다.
2. 선택 결과를 이미지/영상으로 분리한다.
3. 이미지는 선택 직후 자동으로 `1.5초(1500ms)` 영상 변환을 시작한다.
4. 영상은 동시에 여러 편집 화면을 띄우지 않고, 1개씩 순차적으로 처리한다.
5. 전체 처리 중 진행 상태(예: n/m)와 실패 항목을 사용자에게 안내한다.
6. **갤러리에서 사용자가 선택한 순서**를 최종 등록 순서로 보존한다.
7. 실제 처리 길이와 표시 텍스트를 분리해, 표시 문구는 계속 `1s`로 유지한다.
   - 내부 처리 수치(`targetDurationMs`/`normalizedDurationMs`)는 1500ms 기준으로 기록

### 비기능 요구사항
1. 처리 중 중복 탭/중복 실행 방지 (락 플래그).
2. 실패 항목이 있어도 나머지 항목은 계속 처리 (best-effort).
3. 처리 완료 후 현재 앨범 목록 1회 갱신.
4. 대량 선택 시 UI 멈춤 최소화(작업 단위 await, 토스트/배너 최소화).

## 4) UX/플로우 설계

### A. 진입
1. 사용자가 외부 가져오기 버튼 탭
2. 다중 선택 가능한 피커 호출

### B. 선택 결과 분류
1. `selected[]`를 확장자/메타 기준으로 이미지/영상 분류
2. `imageQueue[]`, `videoQueue[]` 생성

### C. 이미지 처리(즉시 변환)
1. `imageQueue`를 순회
2. 각 파일마다 `convertPhotoToVideo(path, currentAlbum, targetDurationMs: 1500)` 실행(또는 정책 상수 주입)
3. 실패 시 실패 목록에 기록 후 다음 항목 진행

### D. 영상 처리(순차 선택)
1. `videoQueue`를 앞에서부터 1개씩 처리
2. 각 영상마다 `ClipExtractorScreen` 진입 → 저장/취소 결과 수신
3. 현재 영상 처리가 끝나야 다음 영상으로 이동

### E. 종료
1. 전체 큐 완료 후 앨범 목록 새로고침
2. 요약 메시지 표시
   - 성공 이미지 수
   - 성공 영상 수
   - 실패/취소 수

## 5) 기술 설계

### 5.1 피커 전략
- 1안(권장): `file_picker`의 `pickFiles(allowMultiple: true)`를 사용하여 이미지/영상 동시 다중 선택
  - 장점: 하나의 다이얼로그에서 대량 선택 가능
  - 단점: 플랫폼별 MIME/확장자 판별 보강 필요
- 2안(대안): `image_picker`의 `pickMultiImage()` + 영상 단일 선택 반복
  - 장점: 이미지 선택 UX 단순
  - 단점: 사용자 요구인 "한 번에 여러 개 선택" 충족도가 낮음

> 결론: 사용자 요구에 맞춰 **1안 채택**.

### 5.2 처리 오케스트레이션
- 신규 비공개 메서드 제안 (`lib/main.dart`)
  - `_pickMediaBatch()` : 다중 선택 + 분류 + 전체 실행 진입
  - `_processImageQueue(List<String> paths)` : 이미지 일괄 변환
  - `_processVideoQueueSequentially(List<String> paths)` : 영상 순차 추출
  - `_isProcessingImportBatch` 플래그로 재진입 방지

### 5.2.1 선택 순서 보존 전략 (추가)
- 핵심 원칙: 피커에서 받은 원본 리스트 순서를 `selectionOrder`로 간주한다.
- 구현 방식:
  1. 피커 결과를 그대로 `selectedItems`(원본 순서 유지 리스트)로 보관
  2. 각 항목에 `orderIndex`(0..n-1)를 부여한 뒤 내부 모델로 관리
  3. 이미지/영상 분리 시에도 **정렬 키를 `orderIndex`로 유지**
  4. 처리 파이프라인은 반드시 `orderIndex` 오름차순으로만 소비
- 권장 내부 모델(문서 레벨):
  - `PendingImportItem(path, mediaType, orderIndex)`
- 처리 정책:
  - 정책 A(권장): 전체를 `orderIndex` 기준 단일 큐로 순차 처리
    - 장점: 사용자가 고른 순서와 실제 등록 순서 100% 일치
  - 정책 B(기존안 확장): 이미지 선처리 + 영상 후처리
    - 장점: 이미지 즉시 변환 체감 속도 우수
    - 단점: 혼합 선택 시 사용자 선택 순서와 등록 순서가 달라질 수 있음

> 최종 권장: 순서 중요도가 높으므로 정책 A를 기본 채택하고,
> 필요 시 설정 플래그로 정책 B를 실험 옵션으로 둔다.

### 5.3 예외/오류 처리
- 파일 없음/권한 취소: 즉시 return
- 지원하지 않는 포맷: 스킵 + 실패 목록 기록
- 변환 실패/추출 실패: 개별 실패 처리 후 진행 지속
- 화면 dispose 상태: `mounted` 확인 후 UI 업데이트
- 순서 보존 무결성: 실패/취소 항목도 `orderIndex` 로그를 남겨 추적 가능하게 유지

### 5.4 사용자 피드백
- 시작: "가져오기 처리 중" 상태 표시
- 중간: 필요 시 경량 진행 텍스트(`x/y`)만 갱신
- 완료: 요약 토스트 1회
- 요약 문구/로그의 시간 표시는 현재 정책상 `1s`로 고정한다.

### 5.5 구현 포인트(표기/길이 분리)
- 권장 구현: 실제 변환 값은 `1500ms` 상수(예: `kTargetPhotoDurationMs`)를 사용.
- 표시 전용 값은 기존 `kTargetClipSec` 또는 별도 라벨 상수로 `1`을 유지.
- 플랫폼 채널도 가능하면 밀리초 단위로 전달해 1.5초를 명시적으로 보장한다.

### 5.6 수치 가정(목표/가드) 정렬
- 목표: 이미지 변환/영상 추출 산출물의 `normalizedDurationMs`가 1500ms 근방이어야 함
- 가드: 오차가 임계치 초과 시 실패로 집계하고 수동 원인 분석 수행
- 요약: 사용자 노출 문구/요약 토스트는 `1s`만 사용

## 7) 호환성/리스크
1. **리스크:** 대량 이미지 변환 시 처리 시간 증가
   - **대응:** 변환 루프 중 최소 UI 업데이트, 완료 요약만 강조
2. **리스크:** 영상 연속 처리 중 뒤로가기/취소 케이스
   - **대응:** 항목별 취소를 정상 결과(취소 카운트)로 집계하고 다음 항목 진행
3. **리스크:** 파일 확장자만으로 타입 판별 시 오탐
   - **대응:** MIME 우선 + 확장자 fallback
4. **리스크:** 플랫폼/피커에 따라 반환 순서가 선택 순서와 다를 가능성
   - **대응:**
     - 반환 순서를 기본 신뢰하되, 디버그 로그로 `orderIndex + path`를 기록
     - QA에서 Android/iOS 각각 선택 순서 재현 테스트 수행
     - 불일치 발견 시 대체 피커/커스텀 선택 UI 검토

## 9) 수동 테스트 시나리오
1. 이미지 5장만 선택 → 모두 1.5초 영상 생성 확인(표시 문구는 `1s`)
2. 영상 3개만 선택 → 추출 화면이 1개씩 순차 진입 확인
3. 이미지+영상 혼합 10개 선택 → 이미지 선처리 후 영상 순차 처리 확인
4. 중간 취소/실패 포함 → 나머지 항목 계속 진행 확인
5. 완료 후 앨범 카운트/목록 반영 및 요약 메시지 확인
6. 갤러리에서 의도적으로 역순 선택(예: C→A→B) 후 등록 결과가 동일 순서(C→A→B)인지 확인
7. 혼합 선택(영상-이미지-영상-이미지)에서 최종 등록 순서가 선택 순서와 동일한지 확인
8. 동일 테스트에서 사용자 문구가 `1s`로 유지되는지 QA에서 수동 검증

## 10) 완료 기준(Definition of Done)
- 사용자가 외부 가져오기에서 다중 선택을 1회로 수행할 수 있다.
- 이미지는 자동으로 1.5초(1500ms) 영상 변환이 완료된다.
- 영상은 동시에 열리지 않고 순차적으로 추출/저장 흐름이 진행된다.
- 실패 항목이 있어도 전체 프로세스가 중단되지 않는다.
- 처리 완료 후 라이브러리 상태가 정확히 갱신된다.
- 사용자가 갤러리에서 선택한 순서와 앱 내 최종 등록 순서가 일치한다.
- 사용자 노출 문구(버튼/토스트/라벨)는 항상 `1s`로 유지된다.
- 길이 정합(정상 항목 기준) 검증 값은 1500ms 또는 허용오차 범위여야 한다.
