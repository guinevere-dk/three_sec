# iOS(App Store) 출시 실행 계획 v1

## 0) 목적
- 현재 안드로이드 출시 성공 상태를 기반으로, **아이폰 실기기 없이도** iOS 출시를 완료한다.
- 목표: TestFlight 내부/외부 테스트를 거쳐 App Store 심사 승인 및 첫 배포 완료.

## 1) 전제 및 제약
- 현 상태: Android는 이미 운영 중.
- 제약: 개인 아이폰 없음.
- 필수 조건:
  - Apple Developer Program(유료) 가입 완료
  - macOS 기반 빌드 환경 확보(로컬 Mac 또는 클라우드 Mac/CI)
  - iOS 심사 대응용 연락 가능 상태(이메일/전화)

---

## 2) 전략 요약 (아이폰 없음 대응)

### 전략 A (권장): 클라우드 빌드 + 외부 iOS 테스터 활용
- Codemagic / Bitrise / GitHub Actions + macOS 러너로 ipa 아카이브 생성
- App Store Connect 업로드 후 TestFlight 배포
- 실제 기기 검증은 내부 팀원/지인/테스터(아이폰 보유자)로 커버
- 장점: 본인 실기기 없이도 실제 사용자 환경 검증 가능

### 전략 B: 단기 실기기 확보(대여/중고) + 병행
- 최소 1대(iOS 최신) 단기 확보하여 핵심 플로우 직접 검증
- 장점: 심사 리젝 포인트를 사전에 빠르게 줄임

> 결론: **A를 기본 경로로 진행**, 가능하면 B를 병행해 리스크를 낮춘다.

---

## 3) 릴리스 단계별 실행 계획

## Phase 1. 계정/인프라 준비 (D1~D2)
- [ ] Apple Developer Program 활성화 확인
- [ ] App Store Connect 앱 레코드 생성 (번들 ID/앱명/카테고리)
- [ ] iOS Bundle Identifier 확정 (`ios/Runner` 설정과 일치)
- [ ] 인증서/프로비저닝 자동화 방식 확정
  - 권장: Fastlane Match 또는 CI-managed signing
- [ ] 배포 파이프라인 구성
  - `flutter build ipa --release` 실행 가능한 CI 환경 구성

### 완료 기준 (DoD)
- CI에서 서명 포함 ipa 빌드 성공
- App Store Connect로 빌드 업로드 1회 성공

## Phase 2. iOS 품질 안정화 (D3~D6)
- [ ] iOS 전용 권한 문구 점검
  - 카메라/사진/마이크 사용 시 `Info.plist` 설명 문자열 명확화
- [ ] 핵심 기능 스모크 테스트 시나리오 정의
  - 로그인
  - 촬영/가져오기
  - 편집/내보내기
  - 클라우드 백업/복원(해당 시)
  - 결제/구독(해당 시)
- [ ] 크래시/ANR 대응 지표 수집 체계 연결
  - Firebase Crashlytics 등
- [ ] iOS UI/레이아웃 최소 호환성 점검
  - 노치/세이프에어리어/다크모드/작은 화면 대응

### 완료 기준 (DoD)
- P0/P1 결함 0건
- 테스트 시나리오 통과율 100%

## Phase 3. TestFlight 운영 (D7~D10)
- [ ] 내부 테스터(팀/지인) 배포
- [ ] 외부 테스터 그룹 구성(최소 5~10명 권장)
- [ ] 피드백 수집 템플릿 운영
  - 재현 단계, 기기 모델, iOS 버전, 스크린샷/영상
- [ ] 수정 릴리즈 최소 1~2회 반복

### 완료 기준 (DoD)
- 외부 테스터에서 핵심 여정 회귀 이슈 없음
- 심사 제출 가능한 릴리즈 노트/변경사항 정리 완료

## Phase 4. App Store 메타데이터/정책 준비 (D8~D11, 병행)
- [ ] 앱 설명(국문/영문), 키워드, 카테고리 확정
- [ ] 스크린샷(6.7" / 6.5" 등 필수 사이즈) 준비
- [ ] 개인정보 처리방침 URL, 지원 URL 준비
- [ ] App Privacy(데이터 수집/추적) 정확 기입
- [ ] 연령 등급/콘텐츠 권리(음원/영상 소스) 확인
- [ ] 계정 삭제 기능 요구사항(해당 시) 충족 여부 확인

### 완료 기준 (DoD)
- 심사 입력 항목 누락 0건
- 정책 리젝 가능성이 높은 항목 사전 점검 완료

## Phase 5. 심사 제출 및 론칭 (D12~D14)
- [ ] Release Candidate 빌드 고정
- [ ] App Review 제출
- [ ] 리젝 시 대응 SLA 수립
  - 24시간 내 1차 회신
  - 48시간 내 수정 빌드 재제출
- [ ] 승인 후 수동/자동 출시 전략 선택
  - 권장: 수동 출시(모니터링 후 오픈)

### 완료 기준 (DoD)
- App Store 승인 완료
- 프로덕션 배포 후 24시간 모니터링 완료

## 4) 체크리스트 (아이폰 없는 상황 특화)
- [ ] iPhone 실기기 테스트를 외부 테스터로 대체하는 운영 계획이 문서화됨
- [ ] 최소 2개 이상 iOS 버전(최신/직전)에서 테스트 피드백 확보
- [ ] 최소 3개 이상 기기 폼팩터(미니/일반/맥스급) 피드백 확보
- [ ] 심사 계정 필요 기능(로그인형 앱이면 리뷰 계정 제공) 준비

## 5) 리스크 및 대응
- 리스크: 실기기 미보유로 UI/권한/성능 이슈 늦게 발견
  - 대응: TestFlight 외부 테스터 확대, 체크리스트 기반 회귀
- 리스크: 결제/구독 심사 리젝
  - 대응: 구독 약관/복원 동작/가격표시/해지 경로 사전 점검
- 리스크: 개인정보/추적 문구 불일치
  - 대응: App Privacy와 실제 SDK 수집 항목 일치 검증
- 리스크: 심사 대응 지연
  - 대응: 제출 전 FAQ/리뷰 노트/데모 영상 사전 준비

## 6) 2주 실행 일정(예시)
- 1주차
  - D1~D2: 계정/CI/서명/업로드 파이프라인 구축
  - D3~D5: iOS 품질 점검 및 수정
  - D6~D7: 내부 TestFlight
- 2주차
  - D8~D10: 외부 TestFlight + 버그픽스
  - D10~D11: 메타데이터/정책 항목 완료
  - D12~D14: 심사 제출/대응/출시

---

## 7) 즉시 실행 액션 (오늘)
1. Apple Developer Program/App Store Connect 상태 최종 확인
2. CI에서 iOS `ipa` 업로드 1회 성공시키기
3. TestFlight 내부 테스터 3명 이상 초대
4. 심사용 메타데이터 초안(설명/스크린샷/개인정보) 작성 시작

이 계획은 **아이폰 미보유** 상황을 전제로 한 최소 리스크 경로다. 핵심은 “클라우드 빌드 자동화 + TestFlight 실기기 피드백 루프”를 빠르게 고정하는 것이다.
