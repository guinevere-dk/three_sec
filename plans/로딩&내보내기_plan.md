1. 프로젝트 준비중...은 현재 진행상황을 알 수 있게 띄워주고, 준비중이 끝나고 내보내기 완료될때까지 중간에 비어있는 Term이 있다. 이거는 내보내는 중 표시가 되어야할 것 같다. 상단 제목표시줄에 백그라운드로 내보내는 중 표시를 띄워주거나 뭔가 다른 표현이 있어야할 것 같다.

2. 내보내기 완료 후 다시 갤러리로 이동해서 영상을 확인해야하는 불편함이 있다. 내보내기 완료 후 즉시 합쳐진 영상을 틀어줄 수는 없나? 갤러리 안에 있는 영상이라 재생시킬 수 없나?
# 로딩 & 내보내기 개선 실행 계획 (벤치마킹 기반)

## 0) 문서 목적

사용자 피드백으로 제시된 두 가지 항목을, **타사 숏폼 편집 앱의 일반 패턴**에 맞춰 실행 가능한 구현 계획으로 정리한다.

1) 프로젝트 준비문구와 내보내기 진행이 단절되어 있어 중간 구간이 텅 비는 현상
2) 내보내기 완료 후 갤러리로만 이동되어 즉시 재생이 불가능한 동선

현재 기준으로 본 이슈는 `프로젝트 준비 중...`와 `내보내기 진행/완료` UX가 분리되어 있는 것에서 시작된다. 앱은 실제로 `프로젝트 준비` 단계(_isPreparingProject)와 `내보내기`가 분리되어 있고, [`main.dart`](lib/main.dart:1020)에서는 준비/가져오기만 오버레이로 표시하고 있어 실제 병합/내보내기 엔진 구간의 상태 노출이 없다.

## 1) 벤치마킹 관찰(외부 앱 패턴)

### 1-1) 타사 공통 패턴

- **TikTok/릴스 계열(공통 패턴)**
  - 내보내기 버튼 즉시 눌렀을 때: 상단/중앙 상주 배너로 `Uploading/Converting` 상태 표시
  - 진행이 길어질 경우 `예상시간`, `완료` 문구를 스텝 단위로 변경
  - 완료 시에는 즉시 미리보기 썸네일/플레이어 진입 동선 제공

- **CapCut/인스타 릴스 편집형 앱 경향**
  - 내부 저장 완료/갤러리 저장 완료 전까지는 작업 종료 처리되지 않음(빈 텍스트 미표시)
  - 완료 토스트 + 재생 버튼/공유 버튼 동시 노출

- **AI 편집기형 앱 경향**
  - 백그라운드 실행 텍스트(예: `내보내는 중`, `임시 저장 중`, `갤러리 이동 중`)를 2~3개 상태로 구간화
  - 사용자가 완료 후 바로 미리보기를 누를 수 있도록 앱 내 재생 위젯을 제공

### 1-2) 우리 코드에서 재활용 가능한 자산

- 완료 시점에서 이미 파일 경로를 반환하고 있음: [`VideoManager.exportVlog`](lib/managers/video_manager.dart:1740)
  - 플랫폼 반환값을 `result`로 받아 저장하고, 로그에는 결과값만 기록됨
  - 내부 저장 경로(`vlog_{timestamp}.mp4`)를 생성하는 구간도 존재
- 완료 동작 중 갤러리 저장이 즉시 수행됨: [`Gal.putVideo`](lib/managers/video_manager.dart:1959)
- 앱 내 미리보기 컴포넌트가 이미 존재함: [`ResultPreviewWidget`](lib/widgets/video_widgets.dart:140)
  - 다만 현재 호출부는 없음

## 2) 목표 상태(요구사항)

### 2-1) 상태 표현 목표

`프로젝트 준비 중...`와 별개로 **내보내기 전용 상태 라벨**을 제공한다.

- `준비중` (현재 `_isPreparingProject` 활성 구간)
- `내보내는 중` (Native 병합/렌더링 시작~종료)
- `갤러리 저장 중` (Gal 저장 완료 대기)
- `내보내기 완료` (완료 + 즉시 재생 버튼 노출)

> 기존 오버레이 구조는 [`main.dart`](lib/main.dart:1073)에 이미 존재하므로, 텍스트만 분기해 확장한다.

### 2-2) 완료 동선 목표

- 현재: 내보내기 성공 시 토스트만 노출, 사용자는 갤러리 이동 필요
- 목표: 내보내기 성공 즉시 앱 내 결과 재생 화면 오픈(또는 바텀시트) + 공유/편집 이어가기
  - 입력으로는 `exportVlog`의 반환 경로를 사용
  - 갤러리 이동은 보조 동선으로 유지

## 3) 기술/구조 변경안

### 3-1) 상태 모델 추가(권장)

`lib/main.dart`와 공유 가능한 상위 상태를 하나 둔다.

- `ExportUiState`
  - `stage: idle|preparing|rendering|saving|done|error`
  - `operationLabel` : 사용자 노출 문구
  - `resultPath` : 완료 시 재생 대상 파일 경로
  - `sessionId` : 디버그 상호 추적

- 상태 변경점
  - `_isPreparingProject`는 유지하되, 내보내기 상태를 별도로 노출
  - 기존 import/로딩 오버레이(`ValueListenableBuilder`)와 병렬 처리
    - 현재는 import 상태만 바인딩되어 있으므로, **내보내기 전용 라벨이 overlay 텍스트로 교체/병행**

### 3-2) 코드 연동 포인트

- `main.dart`
  - `_handleMerge` 시작/종료에서 상태 전환
  - 무료 사용자 분기, 편집 분기(`VideoEditScreen`) 모두에서 동일 상태를 사용
  - 현재 텍스트 라벨이 `프로젝트 준비 중...`인지, `가져오기 처리 중`인지만 보이므로 이를 `export` 라벨과 통합 처리

- `video_edit_screen.dart`
  - [`_performNativeExport`](lib/screens/video_edit_screen.dart:4844)에서 로딩 다이얼로그 닫힘 이후,
    기존에는 `네비게이터 pop`으로 반환만 수행
  - 완료 시 `resultPath`를 상위로 반환 또는 즉시 미리보기 라우팅 가능하도록 콜백/결과 포맷 확장

- `VideoManager.exportVlog`
  - [`exportVlog`](lib/managers/video_manager.dart:1740)에서 `result`가 null인지 판단한 결과만 현재 소비
  - 동일 API 유지 + `merge_session` 로그를 결과 상태와 연결

- 결과 미리보기
  - [`ResultPreviewWidget`](lib/widgets/video_widgets.dart:140) 신규 진입점 추가

## 4) 단계별 실행 계획

### Phase 1 (빠른 완성, 1~2일)

#### 1. 상태 바(로딩) 개선

- 구현 항목
  - `main.dart` 오버레이에서 `프로젝트 준비 중...`만이 아닌
    - `내보내는 중`
    - `갤러리 저장 중`
    를 문구 분기
  - 라이프사이클을 표시하는 문구가 **비어 있지 않게** 유지

- 수락 기준
  - 병합 준비 완료 후 결과 반환 전까지 1개 이상의 적절한 문구가 항상 노출
  - 내보내기 실행 중 최상단 타이틀 표시줄/오버레이에서 `빈 Term` 미표시

#### 2. 내보내기 완료 즉시 미리보기 전환

- 구현 항목
  - free 경로(`_handleMerge` 내 `exportVlog` 호출) 성공 시 즉시 `resultPath` 기반 라우팅
  - standard+ 경로는 `VideoEditScreen` 팝업 결과를 유지하면서, 성공시 `resultPath`를 받아 동일 플레이어로 진입

- 수락 기준
  - 내보내기 성공 시 토스트 + 1회 클릭으로 앱 내 재생 화면 접근 가능
  - 갤러리로 이동하는 별도 동선은 선택 동작으로 유지

### Phase 2 (안정화, 3~4일)

#### 3. 타사 패턴 반영한 완료 플로우 정비

- 구현 항목
  - 완료 화면 또는 다이얼로그에 `공유`, `다시 편집`, `갤러리 열기` 버튼 3종 구성
  - 실패/권한 실패 케이스별 재시도 CTA 강화

- 수락 기준
  - 실패 원인 분류(권한/공간/일시적 오류/강제종료)에 따라 액션 분기
  - 재시도 유저 행동이 즉시 가능

#### 4. 로깅 강화

- 구현 항목
  - 내보내기 상태별 로그 라벨
    - `state=preparing|rendering|saving|done|error`
  - `sessionId`(`mergeSessionId`/`mergeTraceId`)를 기반으로 단계별 소요시간 집계

- 수락 기준
  - 최소 1회 내보내기 세션에서 상태 변화 이벤트 4개 이상 수집(준비, 렌더, 저장, 완료)

### Phase 3 (운영 품질)

#### 5. 성능 및 실패 복구 지표 반영

- KPI
  - 내보내기 시작→완료 소요시간 (p50/p95)
  - 토스트 노출 대기시간(사용자 체감)
  - 내보내기 성공률(네트워크/권한/강제종료 분리)

- 수락 기준
  - p95 가시적으로 기준치 이하(초기엔 임시 허용치 설정)
  - 권한 거부·공간부족·강제종료 케이스가 각 메시지로 구분되어 표시

## 9) 신규 피드백 반영 요구사항 (요약)

최근 세션 피드백 기준, 우선 반영 방향을 다음과 같이 정리한다.

### 9-1) 사용자 요청사항(필수 개선)

1. **프로젝트 준비 진행도 노출 강화** (`프로젝트 준비중..`)
   - 현재 "몇 개 파일 준비 완료"인지 표시
   - 목표: `n/N개 준비 완료` 형태로 직관적 진행 수치 제공
   - 예시: `프로젝트 준비중… (3/5개 완료)`

2. **내보내기 렌더링 진행률 노출** (`내보내기 렌더링중..`)
   - 현재 병합/렌더링 진행 퍼센트 표시
   - 목표: `내보내는 중… 42%` 와 같이 단계별 수치 메시지
   - 네이티브에서 실제 진행률을 받을 수 없을 경우
     - 단계 기반 진행(예: 25/50/75/100)으로 즉시 표시 보장

3. **완료 미리보기 UI 형태 개선**
   - 전체 화면 강제 전환 대신, 결과 재생 화면을 모달/팝업처럼 표시
   - 배경은 유지하고 오버레이 중심으로 재생/공유/갤러리 이동 제어
   - 사용자가 즉시 닫고 작업 화면으로 되돌아갈 수 있게 하기

4. **취소 UX 추가**
   - `준비 중` 또는 `렌더링 중` 상태에서 취소 가능
   - 취소 시 리소스 정리 및 상태 라벨 즉시 정리
   - 진행 취소 후 사용자가 다음 액션(재시도/종료)을 선택 가능하도록 유도

5. **백그라운드/뒤로가기 인터랙션 제어**
   - 준비/렌더링 중에 뒤로가기로 아래 화면 조작이 되는 현재 동작 수정
   - 가이드:
     - A안: 뒤로가기를 일시적으로 차단(로딩 다이얼로그/`WillPopScope`)
     - B안: 뒤로가기는 앱 백그라운드 또는 동작 중단 확인 다이얼로그만 허용
     - A/B 중 1개를 우선 채택해 오동작 방지

### 9-2) 벤치마킹 포인트(외부 앱 기준)

- **CapCut / KineMaster 계열**
  - 준비 단계를 파일 단위로 분할 노출하고, 진행률이 없는 구간은 단계 텍스트로 대체
  - 렌더 진행 중에도 취소/일시정지/백그라운드 처리 동선이 제공되는 편
  - 완료 즉시 미리보기/공유/갤러리 이동을 동일 화면에서 노출

- **TikTok/Instagram 계열**
  - 업로드·렌더 진행 상태는 상단 배너로 지속 노출
  - 완료 이전에는 백스테이프를 제한해 상태 훼손을 방지하고, 완료 후에는 명확한 CTA(재생/공유/편집)를 제공

- **VN/클립 기반 AI 편집기류**
  - 렌더링이 길어지면 단계별 메시지(`프레임 합성`, `색보정`, `오디오 렌더`)로 사용 인식 강화
  - 완료 팝업을 강제 오버레이화해 화면 전환 비용을 낮춤

### 9-3) 추가로 권장되는 개선(우선순위 상향)

1. **진행률 소스 설계 개선**
   - 단기: clip 기반 가중치 진행률(파일 수 기반) + 단계 별 퍼센트 고정 구간으로 우선 구현
   - 중기: 네이티브 메서드 채널에서 `progress` 콜백 수신 시 실측 퍼센트 적용

2. **오퍼레이션 캔슬 토큰/세션 종료 전략**
   - `_handleMerge`와 네이티브 추출/병합 경로에 취소 토큰 전달
   - 취소 요청시 파일 임시본 정리, 저장 반려 방지, 에러 메시지 일관화

3. **완료 플로우 고정 레이아웃**
   - 오버레이 닫힘 없이 `재생(미리보기)` `공유` `갤러리 이동` 버튼을 1차 표시
   - 재생/공유 실패 시 토스트 + 재시도 버튼 자동 노출

4. **터치 인터랙션 블로킹/모드 전환 정책**
   - 렌더링 중에는 상위 화면의 Gesture 막기 (`AbsorbPointer` 또는 `IgnorePointer` 레이어)
   - 앱 상태 전이(백그라운드/포그라운드) 이벤트 로깅 후 복귀 시 마지막 상태 복원

5. **로그 기반 체감 UX 측정 지표(권장)**
   - `preparing/rendering/saving/done/error` 전환 시각 모두 수집
   - 시작→완료 p50/p95, 취소율, 오류코드별 재시도 성공률 추적

### 9-4) 구현 우선순위 제안

- **P0 (즉시 반영)**: 5개 핵심 UX 보강의 최소 가시성(진행 텍스트/퍼센트, 취소 진입, 뒤로가기 제어, 모달형 완료 미리보기)
- **P1 (단기)**: 진행률 신뢰성 향상(네이티브 진행 콜백 연동), 로그 보강
- **P2 (중기)**: 진행 단계 텍스트/아이콘 표준화, 에러/권한 세분화 CTA

## 10) 반영 계획(2차 스프린트)

### Phase 2-A (1회 릴리즈 단위)

1. `프로젝트 준비중..`에 완료 파일 수 표시 적용
2. `내보내는 중..`에 가짜/단계 기반 퍼센트 바 우선 적용
3. `캔슬` 버튼을 준비/렌더링 구간에 추가(세션 단위 중단 UX)
4. 완료 미리보기 모달/팝업 형태로 전환

### Phase 2-B (추가 정합성)

1. 뒤로가기 제어 + 터치 블로킹 레이어 적용
2. 네이티브에서 실제 진행률/상태 이벤트 수신 시 정밀 퍼센트로 대체
3. 세션 로그에 퍼센트/단계/취소 이벤트까지 포함
